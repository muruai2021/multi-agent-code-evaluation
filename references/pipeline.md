## 详细阶段说明

### Phase 1: 任务分析

**主编操作**:
1. 解析评估请求，提取关键信息：
   - 需要评估的文件列表
   - 代码语言和框架
   - 评估深度（简要/详细）
   - 特殊关注点
2. 确定需要调度的 Evaluator
3. 制定评估计划

**评估计划模板**:
```markdown
## Evaluation Plan

- **Files**: 8 files
- **Language**: TypeScript
- **Framework**: Express.js
- **Depth**: Detailed
- **Evaluators Required**:
  - [x] Completeness Evaluator
  - [x] Correctness Evaluator
  - [x] Usability Evaluator
  - [x] Security Evaluator
  - [x] Maintainability Evaluator
```

### Phase 2: 五维并行评估

**执行策略**: 所有 Evaluator 同时开始评估

**评分权重矩阵**:

| Evaluator | 权重 | 核心评估点 |
|-----------|------|------------|
| Completeness | 25% | 功能覆盖、边界处理、错误处理 |
| Correctness | 25% | 逻辑正确、算法准确、类型安全 |
| Usability | 20% | API 设计、可读性、文档 |
| Security | 20% | 漏洞检测、认证授权、数据保护 |
| Maintainability | 10% | 代码结构、模块化、可测试性 |

**并发控制**:
- 最大并发 Evaluator 数: 5
- 单个 Evaluator 超时: 5 分钟
- 全部 Evaluator 超时: 10 分钟

### Phase 3: 分数汇总

**主编操作**:
1. 收集所有 Evaluator 评分
2. 计算加权总分：
   ```
   总分 = (完整性×0.25) + (正确性×0.25) + (易用性×0.20) + (安全性×0.20) + (可维护性×0.10)
   ```
3. 确定评估等级（S/A/B/C/D）
4. 生成汇总表

**汇总计算示例**:
```
完整性: 88 × 0.25 = 22.0
正确性: 85 × 0.25 = 21.25
易用性: 82 × 0.20 = 16.4
安全性: 75 × 0.20 = 15.0
可维护性: 78 × 0.10 = 7.8
───────────────────────
总分: 82.45 ≈ 82 (Grade: A)
```

### Phase 4: 报告生成

**报告内容**:
1. 评估概览（五维雷达图）
2. 各项评分详情
3. 发现的问题列表
4. 改进建议
5. 下一步行动计划

## 评估维度详解

### 完整性 (Completeness) - 25%

**评估内容**:
- 功能实现完整性
- 边界条件处理
- 错误场景覆盖
- 数据验证完整性

**检查清单**:
- [ ] 主要功能是否全部实现
- [ ] 空输入是否有处理
- [ ] 极端值是否有处理
- [ ] 异常情况是否捕获
- [ ] 必要验证是否存在

### 正确性 (Correctness) - 25%

**评估内容**:
- 业务逻辑正确性
- 算法实现准确性
- 返回值正确性
- 类型系统正确

**检查清单**:
- [ ] 业务规则是否正确
- [ ] 算法逻辑是否正确
- [ ] 边界条件处理正确
- [ ] 类型注解是否正确
- [ ] 异步处理是否正确

### 易用性 (Usability) - 20%

**评估内容**:
- API 设计质量
- 代码可读性
- 文档完整性
- 代码可发现性

**检查清单**:
- [ ] API 命名是否清晰
- [ ] 参数顺序是否合理
- [ ] 关键代码是否有注释
- [ ] 是否有 README
- [ ] 是否有使用示例

### 安全性 (Security) - 20%

**评估内容**:
- 漏洞检测
- 认证授权
- 数据保护
- 依赖安全

**检查清单**:
- [ ] 是否有 SQL 注入风险
- [ ] 是否有 XSS 风险
- [ ] 认证是否安全
- [ ] 敏感数据是否加密
- [ ] 依赖是否有漏洞

### 可维护性 (Maintainability) - 10%

**评估内容**:
- 代码结构
- 模块化程度
- 可测试性
- 代码复用

**检查清单**:
- [ ] 文件组织是否清晰
- [ ] 模块职责是否单一
- [ ] 是否易于测试
- [ ] 是否有重复代码

## 评分决策

| 等级 | 总分 | 各维度要求 | 行动 |
|------|------|------------|------|
| S | 90-100 | 全部 ≥ 85 | 可作为标杆 |
| A | 75-89 | 全部 ≥ 70 | 可直接使用 |
| B | 60-74 | 无维度 < 55 | 建议改进 |
| C | 45-59 | 无维度 < 40 | 需要改进 |
| D | <45 | 任意 < 40 | 需要重写 |

---

## 错误处理

| 场景 | 处理策略 |
|------|----------|
| 单个 Evaluator 超时 | 使用默认分数（60），继续其他 |
| 部分 Evaluator 失败 | 已完成的保留，失败的使用默认 |
| 全部 Evaluator 失败 | 返回错误，提示用户检查代码 |
| 文件读取失败 | 跳过该文件，从评估范围排除 |
